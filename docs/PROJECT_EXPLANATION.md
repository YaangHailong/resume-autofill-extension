# 项目面试说明文档：简历自动填写助手

这份文档用于面试时介绍项目。建议按“项目背景 -> 架构 -> 核心难点 -> 验证 -> 改进方向”的顺序讲，听起来会比较完整。

## 1. 项目一句话介绍

这是一个 Chrome Manifest V3 浏览器扩展，用来把用户本地维护的结构化简历，自动匹配并填写到招聘网站或申请网页的表单中。

项目强调两个原则：

- 安全：先预览再填写，不自动提交表单，不上传简历数据。
- 通用：不绑定某一个招聘平台，而是通过字段扫描和关键词匹配适配不同网页。

## 2. 为什么做这个项目

求职时经常要在不同平台重复填写简历信息，例如姓名、手机号、邮箱、求职意向、教育经历、工作经历和语言能力等。很多网页的字段结构不同，有的还需要点击“添加教育经历”“添加工作经历”才能出现更多输入框。

这个项目想解决三个问题：

- 降低重复填写简历的时间成本。
- 保留用户确认环节，避免误填。
- 支持动态新增区块，处理多段教育经历、工作经历和语言能力场景。

## 3. 技术栈

- Chrome Manifest V3
- TypeScript
- React
- Vite
- esbuild
- chrome.storage.local
- jsdom 单元测试
- Playwright e2e 测试骨架

## 4. 整体架构

项目分为四个扩展入口：

```text
popup
  用户点击扩展图标后的入口，负责触发扫描当前页、打开模板页。

options page
  简历模板编辑页面，负责录入和保存结构化简历。

content script
  注入到目标网页，负责扫描字段、生成预览面板、执行填写。

background service worker
  扩展后台，负责打开 options 页面和处理基础消息。
```

共享逻辑放在 `src/shared`：

- `types.ts`：定义简历、网页字段、匹配结果、填写计划等类型。
- `resume.ts`：创建默认简历、扁平化简历字段。
- `fieldDictionary.ts`：中文/英文关键词和字段语义词典。
- `matcher.ts`：字段匹配和动态添加计划生成。
- `storage.ts`：封装 `chrome.storage.local`。

## 5. 核心流程

用户操作流程：

```text
填写简历模板
  -> 保存到 chrome.storage.local
  -> 打开目标网页
  -> popup 点击扫描当前页
  -> content script 扫描 DOM
  -> matcher 生成 FillPlan
  -> 页面右上角显示预览面板
  -> 用户确认或调整映射
  -> 自动填写字段
  -> 如有动态区块，点击添加按钮后重新扫描填写
```

## 6. 简历数据设计

简历数据使用结构化 schema，而不是直接解析 PDF 或 DOCX。

原因：

- 自动填写需要稳定字段，例如 `personalInfo.email`、`education.items[0].schoolName`。
- PDF/DOCX 解析误差较大，不适合作为第一版核心。
- 结构化数据便于多条教育经历、工作经历、语言能力的动态填写。

主要类型是：

```text
ResumeProfile
  personalInfo
  jobIntention
  education
  work
  languages
```

填写前会把结构化简历扁平化成 `ResumeFlatField`，例如：

```text
personalInfo.fullName -> 姓名 -> 张三
personalInfo.email -> 邮箱 -> zhangsan@example.com
education.items[0].schoolName -> 学校名称 #1 -> 西安理工大学
work.items[0].responsibilities -> 工作职责 #1 -> 负责课程平台前端开发
```

这样 matcher 可以逐个字段寻找网页控件。

## 7. 网页字段扫描

content script 会扫描：

- `input`
- `textarea`
- `select`
- `checkbox`
- `radio`
- `[contenteditable="true"]`
- `[role="textbox"]`

每个控件会转成 `FieldCandidate`：

```text
selector
kind
labelText
placeholder
name
idAttr
ariaLabel
contextText
options
sectionHint
```

字段标签来源包括：

- `label[for]`
- 包裹式 `label`
- `aria-labelledby`
- 上一个兄弟节点文本
- 父级和祖先节点中的上下文文本

## 8. 匹配算法

匹配算法不是 AI 模型，而是可解释的启发式打分，并加入了本地字段语义层。

每个简历字段和每个网页字段都会计算分数：

- label 命中关键词，分数最高。
- placeholder 命中关键词，分数较高。
- `name` / `id` / `aria-label` 命中关键词，分数中等。
- 邻近文本命中关键词，分数较低。
- 字段类型匹配会加分，例如 email 对 `input[type=email]`。
- section 语境匹配会加分，例如教育字段优先匹配教育经历区域。
- select 的 option 文本和目标值接近也会加分。
- 字段语义会加分，例如把学历、学位归入教育资质语义，再结合更细的学历/学位类型判断。
- 下拉选项会参与语义判断，例如页面标签写“学位”，但选项是“本科/硕士研究生”，会更像学历字段。

最后按阈值分成：

- `confirmed`：确定匹配。
- `needs-review`：需要确认。
- `unmatched`：未匹配。

为了避免两个简历字段填到同一个网页控件，匹配时会维护 `usedCandidateIds`。当前不是简单按模板顺序贪心匹配，而是先计算所有“简历字段 -> 网页字段”的候选分数，再按全局最高分优先分配，降低“学历抢走真正学位字段”这类冲突。

## 9. 动态添加区块

很多网页默认只显示一条教育、工作或语言记录，需要点击“添加教育经历”“添加工作经历”“添加语言能力”才出现更多输入框。

本项目通过 `SectionAddPlan` 处理：

1. 统计简历模板里某个模块有多少条有效数据。
2. 根据当前网页字段估算页面已有几条。
3. 扫描按钮文本和上下文，寻找“添加教育经历”“添加工作经历”“添加语言能力”等按钮。
4. 计算需要点击几次。
5. 用户确认后执行点击。
6. 等待 DOM 更新。
7. 重新扫描字段并生成新的填写计划。

为了防止误识别导致无限点击，单个模块设置了最大添加次数。

## 10. 预览面板设计

预览面板由 content script 注入到页面右上角，并使用 Shadow DOM 隔离样式。

预览面板提供：

- 匹配统计。
- 动态添加计划。
- 每个简历字段对应的网页字段。
- 手动下拉选择目标字段。
- “不填写”选项。
- “重新扫描”按钮。
- “确认并填写”按钮。

用户手动选择的映射会保存为 `UserMappingOverride`，按当前网站 origin 记忆，下一次扫描同一网站时优先使用。

## 11. 隐私与安全设计

这个项目没有后端服务。

安全边界：

- 简历数据只保存到 `chrome.storage.local`。
- 不把简历数据上传到服务器。
- 不自动提交表单。
- 不自动点击下一步。
- 动态添加也需要用户在预览面板确认。
- 对未匹配字段明确显示，不强行填写。

面试时可以强调：因为简历包含手机号、邮箱、教育经历等敏感信息，所以第一版宁可牺牲一点自动化程度，也保留人工确认。

## 12. 构建方案和踩坑

项目使用 Vite 构建 React 页面，但 content script 采用 esbuild 单独打包。

原因是 Vite 多入口构建会进行 chunk 拆分，可能让 content script 顶部出现：

```js
import { ... } from "./storage.js";
```

Chrome 的 manifest content script 更适合声明一个直接可执行的脚本文件。如果 content script 依赖拆分 chunk，加载和路径管理容易出问题。

最终方案：

- `vite build` 构建 popup、options、background。
- `scripts/build-content.cjs` 用 esbuild 把 content script 打包为 IIFE。
- `manifest.json` 中 `content_scripts` 只引用 `assets/content.js`。

这是项目中比较值得讲的工程化细节。

## 17. 面试回答示例

### Q：为什么不用 AI 直接识别网页？

可以回答：

第一版我选择了可解释的规则匹配，因为浏览器扩展要直接操作用户页面，误填成本比较高。当前已经加入本地语义词典来处理“学历/学位”这类近义字段；后续如果接 AI，也只让 AI 处理低置信度字段，并且把结果放进预览面板让用户确认，不会让 AI 直接操作页面。AI 接入准备记录在 `docs/AI_MATCHING_REQUIREMENTS.md`。

### Q：为什么不自动提交？

可以回答：

简历和求职表单包含敏感信息，而且提交前通常还有隐私授权、附件确认等步骤。为了安全，插件只做字段填写，最终提交由用户人工完成。

### Q：怎么处理多条教育或工作经历？

可以回答：

简历模板里的教育经历、工作经历和语言能力都是数组。扫描页面时会识别添加按钮，生成 `SectionAddPlan`，计算需要添加几次。确认后点击添加按钮，等待 DOM 更新，再重新扫描新增字段并填写。

### Q：这个项目最难的点是什么？

可以回答：

难点不是把值写进 input，而是如何在不同网站上找到正确字段，并控制误填风险。我用了字段候选扫描、关键词打分、section 语境、置信度分级、预览确认和用户映射记忆来解决这个问题。

### Q：如果某个网站字段识别错了怎么办？

可以回答：

预览面板允许用户手动选择正确字段。用户的选择会按网站 origin 保存为 override，下次扫描同一个网站会优先使用这个映射。

## 为什么要按新东方表单重构模板和 JSON？

### 现象

原来的模板是通用简历结构，包含基础信息、项目、证书、技能等模块。但新东方页面的实际表单结构是：

```text

求职意向
教育经历
工作经历
语言能力
```

如果继续使用旧模板，很多字段会变成“模板里有、页面上没有”，例如项目和证书；同时页面上的求职意向字段又没有专门的数据入口，导致匹配和演示都不够清晰。

### 原因

自动填写系统里，模板 schema 和目标页面字段越接近，匹配越稳定。旧模板适合通用求职表单，但新东方页面更像固定招聘系统表单，字段名非常明确，例如 `现从事行业`、`期望月薪(税前)`、`语言类型`。这些字段如果仍然塞进通用字段里，会降低匹配可解释性。

### 解决方案

本次把 `ResumeProfile` 重构为 version 2：

```text
personalInfo
jobIntention
education.items[]
work.items[]
languages.items[]
```

并同步更新：

- options 模板编辑页，按页面顺序展示字段。
- `docs/XDF_RESUME_PROFILE_TEMPLATE.json`，作为可导入和面试说明的 JSON 模板。
- `fieldDictionary.ts`，增加求职意向、新东方教育/工作/语言字段关键词。
- `matcher.ts`，动态添加只围绕教育、工作、语言能力三个可重复模块生成计划。
- 单元测试，覆盖求职意向匹配、新东方静态页匹配和动态添加。

## 时间填写框和下拉框为什么点击确认后没有反应？

### 现象

普通文本框可以被自动填写，但新东方页面里的时间选择框、学历/学位/语言等下拉框没有明显变化。预览面板里看起来已经匹配到了字段，但点击“确认并填写”后页面控件不更新。

### 原因

这类控件不是原生 `<select>` 或普通 `<input>`，而是 Phoenix 这类前端组件库渲染出来的自定义控件。页面里能扫描到一个内部 `input`，但这个 `input` 只是组件的交互入口，真正的状态由 React/组件库维护。

原来的填写逻辑只做了：

```text
element.value = 目标值
dispatch input/change
```

这对普通输入框有效，但对自定义下拉/日期组件不够。它们通常需要：

```text
点击控件
打开下拉/日期弹层
点击匹配选项，或输入后按 Enter/失焦
由组件自身更新状态
```

### 解决方案

本次修复把 `fillPlan` 改成异步交互式填写：

1. 普通 `input`、`textarea`、原生 `select` 仍按原方式填写。
2. 如果输入框位于 `.phoenix-select`、`.phoenix-date-picker`、`.ant-select`、`.el-select` 等自定义控件内：
   - 先模拟 `mousedown`、`mouseup`、`click` 打开控件。
   - 在页面弹层里查找和目标值匹配的选项。
   - 找到选项时点击选项，让组件自己更新状态。
   - 找不到选项时，作为时间框/可输入下拉的兜底方案，写入值并触发 `beforeinput`、`input`、`change`、`Enter`、`blur`、`focusout`。
3. 对截图里这种年月选择器单独处理：
   - 从模板值解析出目标年份和月份，例如 `2024-09` -> `2024年`、`9月`。
   - 在弹层中识别 `2026年` 和 `1月` 到 `12月` 的月份面板。
   - 根据目标年份点击左/右年份箭头，直到切到目标年份。
   - 点击目标月份，让组件用自己的事件链更新值。
4. 动态添加后重新扫描，再按新的异步填写流程处理新增字段。

### 测试验证

新增两个回归测试：

- `自定义下拉框会点击匹配的弹层选项`
- `自定义时间框找不到选项时会写入并触发事件`
- `年月选择器会切换年份并点击目标月份`

验证结果：

## 学历和学位这类语义相近字段为什么会匹配错？

### 现象

模板里有 `学历` 和 `学位` 两个字段，但真实招聘网页不一定严格区分这两个词。比如有些页面标签写的是 `学位`，下拉选项却是：

```text
本科
硕士研究生
博士研究生
```

这些选项本质上更像学历，而不是学位。反过来，也可能出现标签写得很模糊，但选项是 `学士`、`硕士`、`博士`，这更像学位。

如果只靠关键词匹配，就会出现两个问题：

- 页面写 `学位`，模板的 `学历` 匹配不上。
- 模板的 `学历` 和 `学位` 都想抢同一个网页字段，导致真正应该填写的字段被占用。

### 原因

原来的匹配逻辑更偏字符串匹配：

```text
模板字段名 / 关键词
  vs
页面 label / placeholder / name / id / 邻近文本
```

这种方式能解决 `邮箱 -> 邮箱`、`手机号 -> 手机号`，但解决不了“字面不一样、语义接近”的情况。

`学历` 和 `学位` 又比较特殊。它们都属于教育资质语义，中文表单里经常混用，但具体含义不同：

```text
学历：本科、硕士研究生、博士研究生
学位：学士、硕士、博士、无学位
```

所以不能简单把 `学历` 和 `学位` 互相当作同义词，否则会造成误填。

### 解决方案

本次把匹配架构优化成三层：

1. 关键词匹配。
   - 继续处理 `姓名`、`邮箱`、`手机号` 这类明确字段。

2. 本地语义匹配。
   - 在 `fieldDictionary.ts` 中增加 `FieldSemantic`。
   - 把模板字段标记为语义类型，例如 `educationLevel -> education_level`、`degree -> academic_degree`。
   - 从网页 label、placeholder、属性、当前值、下拉选项中推断语义。
   - 对 `学历/学位` 增加更细的冲突判断。

3. 全局最高分分配。
   - 不再只按模板顺序一个个贪心匹配。
   - 先计算所有“模板字段 -> 网页字段”的候选分数。
   - 再按最高分优先分配，已经占用的网页字段不会再给其他模板字段。
   - 这样可以避免 `学历` 抢走真正更适合 `学位` 的字段。

针对 `学历/学位`，当前规则是：

```text
如果页面标签写“学位”，但下拉选项是“本科/硕士研究生/博士研究生”
  -> 判定为学历语义更强，可以匹配模板里的学历字段。

如果页面选项是“学士/硕士/博士/无学位”
  -> 判定为学位语义更强，优先匹配模板里的学位字段。
```

### AI 接入思路

AI 不应该替代这三层本地规则，而应该作为兜底：

```text
本地规则匹配
  -> 只把 unmatched / needs-review 字段发给后端 AI
  -> AI 返回映射建议和原因
  -> 合并进 FillPlan
  -> 预览面板让用户确认
  -> 用户确认后才填写
```

接 AI 时需要准备：

- 一个后端服务地址，本地默认是 `http://127.0.0.1:8787/api/resume-field-match`。
- 火山方舟 API key，只能放在后端环境变量 `ARK_API_KEY` 里，不能写进 Chrome 扩展。
- 一个模型配置，例如 `AI_MATCHING_MODEL=doubao-seed-2-0-code-preview-260215`。
- 扩展的 `manifest.json` 增加后端域名的 `host_permissions`。
- 隐私策略：请求 AI 时只发送字段标签、类型、选项、区块信息，尽量不发送手机号、邮箱等真实值。
- options 页面里的“启用 AI 兜底匹配”开关，默认关闭，需要用户主动启用。

具体接入要求已经整理到 `docs/AI_MATCHING_REQUIREMENTS.md`。

- `全局最高分分配避免学历抢走真正的学位字段`

## 字段已经出现在下拉框里，为什么邮箱仍然显示未匹配？

### 现象

在某个简历填写网页中，页面左侧显示 `邮箱*`，右侧是邮箱输入框。插件扫描后，预览面板的下拉框里可以看到“邮箱”这个网页字段，但当前简历里的邮箱字段仍然显示“未匹配”。

这个现象说明：插件已经扫描到了网页字段，但自动匹配阶段没有把简历字段和网页字段绑定起来。

### 原因

这次问题主要有两个原因：

1. 网页是左右两列布局，`邮箱*` 和输入框不是标准的 `<label for="...">` 关系，也不是包裹式 `<label><input /></label>` 关系。
2. 输入框本身可能只有 `placeholder="邮箱"`。原来的算法对 placeholder 命中的加分是 40 分，但进入匹配的最低阈值是 42 分，所以即使“邮箱”看起来很明显，也会因为差 2 分被判为 `unmatched`。

换句话说，下拉框里能看到“邮箱”，代表 DOM 扫描阶段成功；自动显示“未匹配”，代表匹配打分阶段没有达到阈值。

### 解决方案

本次修复分两部分：

1. 增强 DOM label 识别。
   - 在 `src/content/domScanner.ts` 中增加相邻静态文本识别。
   - 当输入框没有标准 label 时，向上查找父级容器，并扫描当前控件左侧或前面的静态文本。
   - 对 `邮箱*` 这类文本清理 `*`、`：` 等符号，得到更干净的 label。

2. 调整匹配打分。
   - 在 `src/shared/matcher.ts` 中增加精确匹配逻辑。
   - 如果 placeholder 精确等于字段关键词，例如 `邮箱`、`姓名`、`电话`，直接给足够分数进入 `confirmed`。
   - 如果字段属性或 label 精确匹配，也比普通包含匹配给更高分。

### 测试验证

新增了两个测试：

- `placeholder 精确等于字段名时能匹配邮箱`
- `左右两列布局的相邻标签能匹配邮箱`

验证结果：

```text
npm run typecheck  通过
npm test           当前已扩展为 20 tests passed
npm run build      通过
```

### 面试时可以怎么说

可以这样解释：

这个问题暴露了自动填写项目里一个典型难点：扫描到字段不等于能准确匹配字段。很多网页并不会使用标准 label，而是用左右两列布局或者自定义组件。我的处理方式是把问题拆成两层：第一层增强 DOM 扫描，让插件能从相邻静态文本里提取 label；第二层优化匹配算法，让 placeholder 精确命中时获得更高置信度。修复后，我补了对应的 DOM fixture 测试，防止同类问题回归。

## 点击确认并填写后为什么跳转到了工信部备案页面？

### 现象

在新东方招聘网页上点击“确认并填写”后，页面跳转到了：

```text
https://beian.miit.gov.cn/#/Integrated/index
```

同时预览面板里显示动态添加计划时，把 `京ICP备05051632号-16` 当成了“添加教育/工作/语言”的按钮。

### 原因

问题出在动态添加按钮识别逻辑。

原来的扫描逻辑会把 `button`、`a`、`[role=button]` 等元素作为候选按钮，并且用“按钮文本 + 祖先上下文文本”判断是否像添加按钮。新东方页面页脚的备案链接本身不是添加按钮，但它的祖先上下文很大，里面包含了页面其他位置的“添加工作经历”“添加语言能力”等文字，于是被误判成动态添加按钮。

确认填写时，插件会先执行动态添加计划，所以实际点击了备案链接，最终跳转到了工信部备案页面。

### 解决方案

修复分三部分：

1. 收紧添加按钮扫描。
   - 只有按钮自身文本、`id`、`name`、`class`、`aria-label`、`title` 里包含 `添加`、`新增`、`add` 等信号时，才作为添加按钮候选。
   - 不再因为祖先上下文里出现“添加”就把链接当按钮。
   - 排除 `备案`、`隐私`、`privacy`、`beian`、`提交`、`下一步` 等风险文本。

2. 支持新东方自定义添加按钮。
   - 新东方的“添加教育经历”“添加工作经历”“添加语言能力”是 `div`，不是标准 `button`。
   - 扫描器新增对 `[id$="_addButton"]`、`[id*="addButton"]` 的支持。

3. 修复动态条数统计。
   - 原来统计某个模块是否有数据时，把每条记录的 `id` 也算作有效内容。
   - 这会导致空的默认模块记录也被认为需要动态添加。
   - 修复后只统计除 `id` 之外的真实字段值。

### 测试验证

新增了基于新东方静态页的回归测试：

- `新东方静态页不会把备案链接识别成动态添加按钮`
- `新东方静态页的自定义表单 label 能用于匹配`
- `新东方静态页动态添加只选择对应模块按钮`
- `手机号组合控件优先匹配真实手机号输入框`

验证结果：

```text
npm run typecheck  通过
npm test           当前已扩展为 20 tests passed
npm run build      通过
```

### 面试时可以怎么说

可以这样解释：

这个问题说明自动化操作里不能只看上下文文本，否则页面页脚、导航、版权区域也可能被误识别为操作按钮。我把“是否是添加按钮”的判断改成以按钮自身文本和属性为主，祖先上下文只用于辅助判断模块归属。同时把真实线上页面保存成静态 fixture，写成回归测试，保证以后不会再点击备案、隐私政策这类非业务链接。

## 新东方页面为什么整体匹配度很低？

### 现象

在新东方招聘网页中，很多字段能在下拉框里看到，但自动匹配结果很差。比如：

- 邮箱、手机号、学校、学历、专业等字段有时没有自动匹配。
- 手机号区域可能错误匹配到“中国大陆”区号控件。
- 教育、工作、语言等动态模块容易出现错误添加计划。

### 原因

新东方页面不是普通原生表单，而是由 `phoenix-*` 自定义组件和多层 `div` 结构组成。它的真实字段 label 通常藏在：

```text
.form-item
  .form-item__title
    .form-item__text
  .form-item__control
    input / textarea / 自定义 select 输入框
```

这导致原来的扫描逻辑存在几个问题：

1. 只依赖标准 `<label for="...">`、包裹式 `<label>` 或相邻文本，无法稳定拿到 `.form-item__text`。
2. 自定义 select 内部的真实可输入节点通常只是一个空 placeholder 的 `input`，如果拿不到 label，就很难判断它代表“学历”“学位”还是“开始时间”。
3. 手机号是组合控件，里面既有“中国大陆”区号，也有真正的手机号输入框。两个控件都在“手机号”区域里，如果只看 label，容易选错。
4. 动态模块里默认对象都有 `id`，原来统计“这个模块是否有内容”时把 `id` 也算作有效内容，导致空模块也可能触发动态添加。

### 解决方案

修复分四部分：

1. 增强自定义表单 label 识别。
   - 在 `src/content/domScanner.ts` 中优先识别 `.form-item` 内部的 `.form-item__text`。
   - 对新东方这类自定义组件，直接把 `.form-item__text` 作为当前控件 label。

2. 扩大相邻文本查找范围。
   - 当字段没有标准 label 时，向上查找更多层父容器。
   - 过滤掉过长文本、提交/保存/取消等非字段文本，减少误判。

3. 优化手机号匹配。
   - 在 `src/shared/matcher.ts` 中增加 `valueShapeScore`。
   - 对包含 7 位以上数字、placeholder 包含 `11位手机号`、字段值像手机号的控件加分。
   - 对 `中国大陆`、`区号`、`国家`、`region` 这类区号控件降分。

4. 修复动态条数统计。
   - 统计模块有效数据时忽略 `id` 字段。
   - 只有除 `id` 之外的真实字段有值，才认为这一条需要动态添加。

### 测试验证

新增了基于新东方静态网页的测试：

- `新东方静态页的自定义表单 label 能用于匹配`
- `手机号组合控件优先匹配真实手机号输入框`
- `新东方静态页动态添加只选择对应模块按钮`

验证结果：

```text
npm run typecheck  通过
npm test           当前已扩展为 20 tests passed
npm run build      通过
```

### 面试时可以怎么说

可以这样解释：

这个问题的本质是招聘网站用了自定义组件，DOM 结构和标准表单不一样。原来的扫描逻辑偏向标准表单，所以在新东方页面上拿不到稳定 label，导致匹配算法没有足够信号。我没有针对某一个字段硬编码，而是增强了通用扫描规则：识别 `.form-item__text`、优化组合控件打分、修复动态模块有效条数统计。这样既解决了新东方页面，也提升了对其他低代码/组件化招聘网页的兼容性。
